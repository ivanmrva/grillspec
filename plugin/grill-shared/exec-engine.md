# exec-engine — shared core for build/verify skills (NOT an interview)

The shared method behind every exec profile. The profile's `SKILL.md` is the entrypoint and names what varies (the task, scope, what it touches); this engine is the judgment that's the same for all of them — the TDD cadence and test layers, the autonomous done-gate, the workflow, and what to decide vs. ask.

**Your output is code and reports in the project source tree** — that, plus your ADRs in the shared `adr/` folder, is the record; there is no side ledger. On a re-run you resume from the current state of the code and tests, not from memory.

**The baseline is the current workspace, nothing else.** Implement, verify, and resume against the **working tree exactly as it exists now** — the spec, code, and tests on disk. **Never** treat **git history** (old commits, `git log`/`git show`, stashes, branches, reflog) as a source of truth to reconstruct from: a removed file is **gone, not a recoverable draft**; "it used to work" is settled by the current code, not a past version; missing prior work is a **blocking gap to escalate**, not something to restore from history. (Routine git *mechanics* — branching/committing per the workflow below, or `git bisect` to localise a regression in `diagnose` — are fine; this fence is about what you treat as the baseline.) Likewise **never read or write outside the project folder** — no parent dirs, home dir, sibling repos, or absolute paths beyond the workspace root — **unless the user explicitly hands you that input**.

## Tight context (efficiency — what keeps a coding agent fast & accurate)
Load only the task, the exact spec IDs it references, and the code it touches — nothing more. A bloated context is slower and less accurate; precision is the discipline.

## Honor the spec — don't reinvent it
**Work from the task's *scoped* references — do NOT load the whole spec.** A coding slice is **construction, not synthesis**: it needs exactly its acceptance criteria, the domain elements it realises, the contracts/schema at its boundary, the architecture boundaries it must respect, and the conventions — that scoped set. You **may pull a specific upstream artifact on demand** if a boundary is unclear, but never ingest all upstream layers: market/GTM/business-model and unrelated requirements are *noise* that degrades focus and accuracy (irrelevant — especially topically-adjacent — context measurably hurts focused work). Scope tight; retrieve just-in-time. (Broad upstream context belonged to the *synthesis* that drew this slice's edges — the architecture and the task breakdown — not to building it.)
Implement to the referenced spec IDs exactly (domain model, contracts, data model, security rules, NFR/ASR decisions). If the spec is wrong, contradictory, or missing for this slice, **STOP and raise it** — record the blocking gap on the task and escalate to re-open the source spec; never silently diverge.

## Edge cases discovered while coding (the deepest last-responsible-moment)
Code reveals edge cases no model anticipated — expected, not a failure. When you hit an **undecided behavior** mid-implementation, **stop, don't guess**: record a `GAP <what's undecided> — <owning area> — UNRESOLVED` on the task and resolve it just-in-time one of three ways — **complete it** (elicit the rule/UX — it lands in the source spec), **mark N/A** for this slice, or **accept a default** (an ADR in `adr/`). The task is implementation-final again only once the gap is resolved. Never encode an invented behavior to keep moving. The same rule covers a **missing or inadequate design-system element** a UI slice needs (a component/variant/token the design system lacks): that is a `GAP … — design-system — UNRESOLVED` raised to the design system — **never bespoke-styled in code to keep moving** (the conventions forbid raw values / ad-hoc styling anyway); resolve it by extending the design system (which then propagates to the prototype + code) or marking it N/A for this slice.

## Incremental + always propagate
On a re-run after a change, **re-touch only the impacted code** (the change set you're given) — don't rewrite green, in-boundary code; re-run the tests and the conformance check on exactly the impacted slices.

## Architecture boundaries are law
Respect bounded-context boundaries, layering, and the dependency rules in your conventions. No cross-boundary shortcuts — long-term architecture is preserved only by enforcing these on **every** task.

## TDD & vertical slices — the red → green → refactor cadence
Build one **minimal vertical slice** at a time, end to end (entrypoint → application → domain → persistence); no horizontal layers, no big-bang. Within a slice, work the tight TDD micro-cycle — **never write production code without a failing test that demands it**:
1. **RED** — write ONE small test for the next bit of observable behavior (from an `AC-` or a test-strategy edge); run it, watch it fail for the right reason. A test you didn't see fail proves nothing. **Record that red run so the project-local exec-gate lets you write the code:** `python3 ${CLAUDE_PLUGIN_ROOT}/tools/gate_exec.py --red --test "<the command that runs this failing test>"` (it re-runs the command, *requires* a non-zero/failing exit, and only then unblocks `src/**` edits). The gate reads the active task from your **`task/T-NNN` branch** (so it's parallel-safe — an AFK wave on per-task branches/worktrees each enforces independently); only if you are *not* on a task branch do you first set it explicitly with `… --start T-NNN`. The gate is the harness-level backstop for "no production code without a failing test" — not a substitute for the cadence, the enforcement of it. (Override for a sanctioned non-TDD edit — a spike, a pure refactor: `GRILLSPEC_GATE_OFF=1`.)
2. **GREEN** — the **minimal** code to pass; the smallest version, not the elegant one — and the smallest *real* implementation, never a fake (see *Production code is production-only* below).
3. **REFACTOR** — only **with the suite green** (never while RED): improve names, structure, and duplication without changing behavior, re-running after each step.

**Don't batch.** Writing all tests then all code is horizontal slicing and produces tests that assert imagined behavior and pass when behavior breaks. One test → one implementation → repeat. Tests assert **observable behavior through the public interface, not internals** (the warning sign of a bad test: it breaks on a behavior-preserving refactor). Each `AC-` → at least one test; an edge discovered here becomes a test (and, if it reveals an undecided rule, a gap).

## Production code is production-only — no fakes in the shipping path
From the **very first line**, `src/` carries **only production behavior** — there is no "early stage" in which fake code is acceptable scaffolding. "Minimal code to pass" means the **smallest *real* implementation**, never a faked one. A stub, mock, in-memory double, hardcoded/canned response, `TODO`-shaped short-circuit, or an *"if the real system isn't wired yet, return something plausible"* branch is **non-production code and never lives in `src/`** — test doubles belong **only** under `tests/`, standing in for a collaborator *inside a test*. A thin-but-real slice (a real call, a real query, a real adapter — however small) is the green; a fake that imitates the behavior is **not done**, even when every test passes against it. That is exactly the trap: green tests against a fake ship a system that is green and does **nothing real**, and the fake outlives the excuse for it.

**An absent dependency is something the test must REVEAL, not something the production code is bent around.** If a dependency (library, service, credential, datastore) isn't available, the slice's test **goes red** — and that red is the **truthful signal** that the prerequisite isn't met yet, not a problem to engineer away. You **never** adapt the production code to stay usable without the dependency, to no-op when it's missing, or to turn the test green in its absence. Red-because-the-dependency-isn't-there is the correct, honest state; you resolve it by **supplying the real dependency**, not by degrading the code that needs it.

**An unconfigured external system is a blocking prerequisite, not a licence to fake it.** When the slice needs a credential, broker, datastore, queue, or third-party that isn't wired yet, do exactly one of: **(a)** drive the **real** dependency through a test container / local double **in the test layer**, while the production adapter stays real; or **(b)** if the real thing genuinely can't run yet, **escalate** — append the provisioning step to `spec/_human-input.md` and the walking-skeleton `bootstrap.md`, mark the task `blocked`, and move to other work. You **never** unblock yourself by embedding the fake in the production path "to get through the early stages." The only sanctioned throwaway is a **spike** (above): it proves shape, lives outside the slice, and is **deleted before** the green slice — it never becomes the implementation.

**No silent scope reduction — the deploy/CI artifacts are production too, held to the same bar.** "Production-only, no-fakes" covers **every** artifact the slice ships, not just `src/`: the **deploy scripts, the CI/CD pipeline stages, the IaC, the migrations, the env-config** are production code, produced **from the very first task** as the design specifies — real, not stubbed, not deferred. The walking skeleton ships a **real deploy of the running system to the first environment of the ratified promotion path** (per `infra-ops/topology.md` — dev, a dynamic preview env, whatever that path begins with) with a smoke test and an automated rollback, plus the CI/CD pipeline the design defines. A required artifact you decline to produce is a **blocking escalation, never a silent omission** — you do **not** "deduce the infra isn't ready" and skip the deploy script, the CI stage, or the migration. Do exactly one of: **(a)** ship the **real** artifact — and if the target environment isn't provisioned yet it **honestly fails to run / stays red**, the truthful signal (resolve it by provisioning, recording the exact step in `_human-input.md` + `bootstrap.md`), never by faking a green deploy; or **(b)** if it genuinely can't be authored yet, **mark the task `blocked` and escalate** the precise missing prerequisite. A fake deploy (an `echo "deploying"`/`exit 0` job, a `# TODO: deploy`, a commented-out or `if: false` step) is **non-production code** exactly like a stubbed function. **A task whose deploy pipeline is incomplete is NOT done** — reporting it done while the pipeline can't actually ship is the precise false-success this rule exists to stop. The cross-stack backstop is `check_deploy_real.py` (the deploy-side sibling of `check_no_fakes.py`); the project's own pipeline test is the per-project check.

## The slice's test set spans every layer it touches — produced WITH the code
A tracer bullet cuts through all layers, so its tests must too. Produce the full set with the slice, per your test strategy's levels:
- **unit** — domain logic, invariants, pure functions (fast, no I/O; the bulk).
- **integration** — the adapters and persistence the slice adds (real DB/queue via a test container or a double); the boundary to an external system. **This is where test containers / `docker-compose` belong — NOT e2e.**
- **contract** — every `API-`/`EVT-` the slice exposes or consumes (schema, versioning), provider- and/or consumer-side.
- **e2e** — the slice's user-observable path end-to-end **against a REAL deployed environment** (the first env of the ratified promotion path — an ephemeral per-PR **preview env** by default, or a reserved e2e/staging env per the test strategy), **never a local `docker-compose`/testcontainers stack** (that's integration mislabelled — it never exercises the deploy, env-config, secrets, or networking that break in prod). One happy path + the load-bearing edges; kept few, they're expensive. Because it hits the deployed system, a **green e2e/smoke is itself the behavioural proof the deploy pipeline actually shipped** — the authoritative deploy check once the env can run; `check_deploy_real.py` is the static backstop for the early window where the env isn't provisioned yet.

Still TDD and one-at-a-time within each level, still behavior through the interface. An `NFR-`/`ASR-` target gets its **evidence test** here, not an assertion. These are the **per-slice (Tier-A)** tests; cross-feature journeys, architecture fitness functions, and NFR-evidence runs are the **system suite (Tier-B)** — owned by the test strategy, seeded by the walking-skeleton task, run in CI.

## What a good review checks (the MR/PR gate, not just CI green)
Before merge, confirm: behavior matches the referenced `AC-` and contracts · architecture **boundaries and dependency rules** intact · tests assert behavior (not internals) and cover the edges · names use the **ubiquitous language** · no dead code, no scope creep beyond the task. A green pipeline is necessary, not sufficient — these are the human-judgment checks it still owes.

## Code lives in the repo, not in spec/
Your code and tests live in the project source tree. The spec is the source of truth you implement to; you never edit it to match your code (if it's wrong for this slice, escalate). Your ADRs — in the shared **`adr/`** folder, prefixed with your area code (`ADR-<AREA>-NNN`) — record genuinely emergent implementation decisions.

## The process bookends are mandatory, not optional steps — tests open every task, the conformance review closes it
The per-task loop has a **fixed shape that is itself part of Done**: it **opens** with the failing tests (you cannot begin implementation until the slice's tests exist and have been **seen to fail** for the right reason) and it **closes** with an independent conformance review that records a `VERDICT: PASS`. Skipping either bookend means the task is **not done**, even if the code looks finished and the suite is green. These steps are not a recommended ordering you may collapse under time pressure — they are the gate. Concretely:
- **Tests-first is enforced by its trace, not by trust.** Every referenced `AC-` must be exercised by a real, behavior-pinning test that is **able to fail** (the mutation gate proves the assertions bite). A task whose tests were back-filled after the code — or never written — shows up as an `AC-` without a failing-capable test, and that is a **red gate**. (Temporal order on its own can't be proven after the fact without leaning on git history, which is **not** a source of truth here — so the *commit-time* proxy is: every `AC-` has a test that genuinely can fail, authored to drive the behavior, not to rubber-stamp it.) **Order *is* observable at tool-call time, and that's where it's enforced live:** the project-local **PreToolUse exec-gate** (`gate_exec.py`, installed by the walking-skeleton) blocks a `src/**` edit until this task's RED was recorded (above), and blocks flipping the record to `status: done` until `check_task_record.py` passes — so the cadence can't quietly collapse mid-session, not just be caught at commit.
- **The conformance review is a real artifact or it didn't happen.** A task is done/mergeable only when a recorded `VERDICT: PASS` for **this `T-`** exists on disk under `spec/10-delivery/verification/`, **produced in a fresh / independent context** — never the implementing loop asserting its own pass. No artifact, or a `FAIL`/missing verdict → not done.

**Both bookends are reconciled mechanically by a per-task Verification Record** — `spec/10-delivery/verification/tasks/T-NNN.md`, an obligation table whose rows are **generated from the task's frozen spec references** (so the bar can't be authored to mark its own homework). Generate it **before** implementation (`python3 ${CLAUDE_PLUGIN_ROOT}/tools/check_task_record.py spec --init T-NNN` — every row `PENDING`), fill each row's evidence as you go, and flip `status:` to `done` only when every row is `PASS`/`N/A`. The checker (`check_task_record.py spec --task T-NNN`) **fails the gate** when a record claims done but an obligation is unmet, a referenced obligation is missing (bar shrunk), the **`deploy` or `tests:layers` gate row is omitted** (a required artifact silently dropped rather than shipped real or marked `N/A — why`), the conformance row has no independent `VERDICT: PASS` on disk, an `AC-`/`API-`/`EVT-` has no passing traceability row (a back-filled, not test-first, test), an evidence path doesn't exist, or a coverage/mutation number is below its bar. An **in-progress** record never blocks — only an unbacked **done-claim** does. The checker reconciles paperwork; it doesn't replace the review's semantic judgment (does the test truly pin the `AC-`).

## Definition of Done (every task)
The **layered test set** green (unit/integration/contract/e2e as the slice warrants) — **every level the test strategy requires for this slice present**, not just the ACs' unit tests — **driven test-first** (every `AC-` has a failing-capable test that drove the code) · conforms to architecture · builds and lints clean · spec-lint clean · referenced NFR/security rules **evidenced** · **the slice's deployable surface ships a real, smoke-verified deploy/CI artifact (or `N/A — no new deployable surface`) — `check_deploy_real.py` clean, no faked/skipped deploy** · **any schema/data change ships a real migration (or `N/A`) — `check_migration_real.py` clean** · traceability updated · **a recorded independent conformance-review `VERDICT: PASS` for this `T-` exists under `spec/10-delivery/verification/`** · **the per-task Verification Record (`check_task_record.py`) is green** (every obligation the task references is evidenced, the `deploy` + `tests:layers` rows carried; `status: done`). Report your status; **never mark done on red**, and **never mark done without the conformance artifact and a green Verification Record**.

## Autonomous done-gate & self-correction loop (AFK execution)
For autonomous work, don't stop at the first failure or hand back a half-done task — **drive it to a fully green gate yourself, fixing your own code**, and stop only on a true blocker.

**The done-gate (run as ONE gate — the same set CI runs):** build · format · lint · type-check · unit · integration · contract · e2e · **every test level the strategy requires for this slice present** · **architecture fitness functions** · spec-lint · the derived-file guard · **a recorded conformance-review `VERDICT: PASS` for this `T-` exists under `spec/10-delivery/verification/`, produced in a fresh / independent context** (code vs spec & arch — not the implementing loop's own say-so) · **the production tree is fake-free** (`check_no_fakes.py` + the no-fakes fitness function — no stub/mock/double/canned-response/unwired-fallback in `src/`) · **the deploy/CI surface is real, not faked or skipped** (`check_deploy_real.py` — no `echo`-only/`exit 0`/`# TODO`/`if: false` deploy; the slice deploys to the first env of the ratified promotion path, or carries `deploy: N/A — why`) · **schema/data changes ship a real migration, not a faked/empty one** (`check_migration_real.py` — no `-- TODO`/empty `operations = []`/DDL-less migration) · **every `AC-` exercised by a passing `@covers AC-`-tagged test that was able to fail** (the tests-first trace, verified against the test source) · traceability updated · **the per-task Verification Record green** (`check_task_record.py --task T-NNN` — every referenced obligation evidenced, the `deploy` + `tests:layers` rows carried, `status: done`). Done only when **all** are green.

**The loop — repeat until fully green or you escalate:** (1) run the **whole** gate, collect **all** failures; (2) diagnose each to root cause; (3) **fix the implementation code**, re-run the whole gate; (4) if the failure set didn't shrink for two consecutive iterations, or you hit the cap, **stop and escalate** — don't thrash, don't reach for a shortcut.

**Anti-cheat invariants — violating any means NOT done, even if everything is green:**
- **Fix the code, never move the goalposts.** Never delete or weaken a test or assertion, lower a coverage threshold, add `skip`/`xfail`/`ignore`, loosen a lint/type/fitness rule, or stub a function to fake a pass. A test changes **only** when the *spec* changed.
- **The spec is the source of truth.** Never edit the spec or an `AC-` to match buggy code; if the spec is wrong or ambiguous for this slice, **escalate**.
- **Green ≠ done.** Passing tests are necessary, not sufficient: the conformance check (judged independently of test colour) must pass, and every `AC-` must be exercised by a real test.
- **Never disable a gate, fitness function, or hook** to make progress.
- **No fakes in the shipping path — production code is production-only.** A stub, mock, in-memory double, canned response, `TODO`-short-circuit, or *unwired-fallback* branch in `src/` means **not done**, even if every test is green — test doubles live only under `tests/`. An external system that isn't configured is a **blocking prerequisite to escalate** (`spec/_human-input.md` + `bootstrap.md`), never a reason to fake it in production; an absent dependency is meant to keep the test **red** until it's supplied, never to be coded around.
- **No silent scope reduction — deploy/CI artifacts are production too.** Never drop a required artifact (a deploy script, a CI/CD stage, a migration, an env-config) by **deducing the environment "isn't ready"**; produce the **real** artifact (it may honestly fail / stay red until the env is provisioned — that's the truthful signal, recorded in `_human-input.md` + `bootstrap.md`), or mark the task `blocked` and escalate. A fake deploy (`echo "deploying"`/`exit 0`, `# TODO: deploy`, a commented-out or `if: false` step) means **not done** — and **omitting** the artifact while reporting the task done is the same cheat, caught by the `deploy` row's required-presence and `check_deploy_real.py`. A **fake or empty migration** (a `-- TODO`, an `operations = []`, a DDL-less `.sql`) is the same class of fake, caught by `check_migration_real.py`. **A task whose deploy pipeline can't actually ship — or whose schema change doesn't actually migrate — is not done.**
- **Clean architecture always wins — never degrade structure to pass a test.** Fix failures **inside the designed boundaries**; you may not cross a context or layer boundary, reach into another module's internals, bypass a port/repository, fatten an interface, duplicate logic, introduce a cycle, or add a hack — even to make a test green. If the clean design genuinely can't satisfy the test/`AC-`, that's a **design/spec signal — escalate**, not a licence to hack.

**Escalate (stop autonomy) only when:** the spec is genuinely ambiguous or contradictory for this slice · a HITL input is truly required (a visual/UX decision, a product/legal/strategy call, an external credential/access) · the clean design can't satisfy the test/`AC-` without crossing a boundary · the loop can't converge after the cap. Append to the batched human-input queue (`spec/_human-input.md`): the task id · the precise blocker · **the actionable step-by-step resolution** (for a credential/access/provisioning block, exactly what to create and where to set it — never the secret value; for a UX or decision block, your **proposed default for the human to ratify**) — detailed enough to clear in one sitting without guessing. Mark the task `blocked`, and move to other AFK-eligible tasks. Never mark done on red; the CI run on the PR is the backstop, not the bar.

## Engineering workflow (standard practice — from conventions)
Per task: **branch off main** (`task/T-NNN-…`, never commit to main) → write the layered tests first → implement → **pre-commit hook runs the enforcers** (spec-lint · format · lint · type · secret-scan · fast tests · the mechanical conformance and fitness functions on the changed scope) → **conventional commit** → **open one MR/PR** → **CI** (build · full tests · full conformance · gate checks) → review → **merge** (auto-merge on green only where the conventions' review policy permits solo-merge; where it mandates ≥2-person review / branch protection, an `afk:eligible` agent parks the merge for a human rather than self-approving) → **deploy to the first environment of the ratified promotion path (per `infra-ops/topology.md`) → e2e/integration pipeline + docs site built**, then promote along that path per the ratified gates. Small, atomic, reversible — the *checks* aren't optional, the *release* decision is the team's.

## House style
Reports and artifacts: tight, structured, prose to clarify not walls; use the ubiquitous language; cite spec IDs. **Prefer a Markdown table whenever the content is even loosely tabular** — anything repeated over a set of items (results, gate checks, failures, IDs, mappings, AC×status) is a table, not a bullet list or prose, because tables are the most scannable form for the reader; reach for other readability structures (fenced code, a small diagram) where they fit better. **Where a table's row defines an ID-keyed record (a traceability row), the ID is the leading column** (the row key), and IDs you cite use their **bare type prefix** — never a context-namespaced `<CTX>-TYPE-NNN`. **Any artifact you write under `spec/` (a verification or operations record) opens with the `<!-- scope: … | excludes: … | format: … -->` header, like every spec file** (code under `src/`+`tests/` does not). When you implement or verify against an external **standard/format/protocol** (a WCAG level, an API/security standard), target its **current / latest stable** version, never a pinned historical one. **A record you write under `spec/` (a verification or operations record), and every ADR, is a timeless source of truth — it states what is true now, never the document's own history.** A reader has no idea it was generated, by what tool or in how many passes, so leave no development trace: never tag content with its edit history (`new` / `newly added` / `now` / `previously` / `no longer` / `expand→contract`), never note what *this* pass added, moved, cut, or deferred, never write `this round` / `as discussed`. The same word stays legitimate as **domain language** (a `new booking`); it is banned only when it narrates the document's own evolution. When you update such a record, **re-author it as a coherent whole** — integrate the change where it belongs, never append it and leave the rest; the result reads as if written in one pass, and emphasis marks meaning by the record's convention, never that something was just added. (A transient run-report back to the orchestrator may of course say what changed — that is not a `spec/` artifact.)

## System friction → plugin feedback, never your artifact
If you hit a defect or gap in **this system itself** — a check that's demonstrably wrong, an instruction you can't satisfy, a conformance rule that contradicts the spec it's checking — **don't silently work around it**; append it via `python3 ${CLAUDE_PLUGIN_ROOT}/tools/plugin_feedback.py --add --kind bug|gap|improvement --component <this skill/the tool> --summary "…" --suggest "…" --evidence <file:line>` (it writes `GRILLSPEC-FEEDBACK.md` at the project root, outside `spec/`). Capture-and-route for the plugin author; never self-patch.

# Deciding vs. asking

What you DECIDE vs. what you escalate is defined once, for every skill, in **`${CLAUDE_PLUGIN_ROOT}/grill-shared/decision-classes.md`**. **Load it and apply it to every fork.** In one line: default hard toward **deciding** what engineering merit settles; raise a **gap** only for a fact the user/org/platform holds that the spec can't carry; pick the conventional option for inconsequential taste.
