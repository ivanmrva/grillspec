# derive-tasks

**Load `${CLAUDE_PLUGIN_ROOT}/grill-shared/derive-engine.md` first and follow it.** This skill applies that method to **task breakdown** — turning a recorded spec into minimal, vertically-sliced, build-ready tasks. A task is rarely one-shot: derive → ask → iterate until each is implementation-final (zero ambiguity), producing slice-specific detail (incl. UX mockups/microcopy) the global spec doesn't cover.

## Method
1. **Start with the walking skeleton.** The first task wires the architecture end-to-end, scaffolds the repo, stands up governance (below), wires the design-token pipeline when there's a UI, **really deploys to the first environment of the ratified promotion path** (per `infra-ops/topology.md` — dev, or a dynamic preview env, whatever that path begins with) **with a smoke test + an automated rollback** (the release path is part of the end-to-end wiring — a real deploy, never a stubbed/`echo`/`exit 0` one), emits the human-prerequisites bootstrap checklist, and seeds the Tier-B system/journey acceptance tests + architecture fitness functions as pending — so they exist before any feature code. The full promotion workflow it wires (which env follows which, and the gate at each hop) is **taken from `infra-ops/cicd.md` + `topology.md`, not invented here**.
2. **Slice vertically.** Cut the first MVP use-case into a tracer bullet: a narrow but COMPLETE path through every layer (schema → domain → api → ui → tests), demoable on its own. Never a horizontal slice of one layer; right-size to one agent session.
3. **Fill every dimension by reference.** For each slice, each applicable dimension (behavior, domain, data, api, ux, a11y, security, nfr, integration, placement, **design**, **prototype-review**, tests, **deploy**, **human-prereq**, dependencies, DoD) is exactly one of: a spec-ID reference, `N/A — why`, or `GAP <what's missing> — UNRESOLVED`. **Fields are ID-keyed and structured, not prose blobs.** The manifest is a **single two-column `field | value` table** — one row per dimension, with title/phase/afk/outcome on their own rows; one uniform table, no dimension escaping to a side block. The two sub-structured fields render their structure **inside their own cell** with `<br>` line-breaks (a markdown cell can't hold a nested pipe-table, so `<br>`-separated lines — **never** a "see table below" split): **behavior** is one `<br>`-line per `UC-` with each of its `AC-`s on a following `<br>`-line (the obligation backbone the Verification Record is generated from); **tests** is **one `<br>`-line per AC inside the single `tests` cell** (never a pointer to a table elsewhere) — each line is `AC-id — tiers (`·`-joined) — test intent`, the **test intent** being the behaviour that pins it (e.g. "widens exactly one rung; rejects narrow"), never a prose blob and never a re-echo of the AC id. (Tagging the actual test with the AC id is a test-file convention owned by `derive-conventions`; the manifest doesn't restate it — the row carries the *intent*, not the tag.) Any cross-reference (another `UC-`, a `JRN-`) is the **bare id**, since only ids the linter sees are enforced. The **ux** dimension references the journey by **`JRN-` id** + the **interaction-states** this slice realises, then points at the slice's **clickable prototype screen** (the exact visual + interaction reference to build against, generated at execution time and reviewed at the `prototype-review` gate before code); the **a11y** dimension names the keyboard path · focus order · contrast · the relevant current-WCAG success criteria (default `N/A — headless` for a non-UI slice); the **design** dimension is `inline` for a simple/CRUD slice (the agent designs as it TDDs) or **`design-first`** for a complex slice (hard algorithm · real concurrency · cross-context saga → its module internals are designed & reviewed before coding); the **prototype-review** dimension is the **task-start gate** at which the just-generated prototype / impl-design artifact is reviewed **before any code** — **auto-AFK** (no human) when it only realises already-ratified `JRN-` journey states, escalating to **HITL (`afk: blocked — visual/UX decision`)** only when it introduces an unratified visual/UX choice (`N/A — headless` for a non-UI slice); the **deploy** dimension names the **deployable surface this slice adds** and the environment(s) it must reach along the ratified promotion path (reference `infra-ops/cicd.md` + `topology.md`) — for the walking skeleton this is the real end-to-end deploy to the first env of that path with smoke + rollback; for a feature slice it is the flag/migration/config it ships into the pipeline, or `N/A — no new deployable surface`. It is **never** `deferred — infra not ready`: an unreachable environment is a `human-prereq` to escalate (provision it), not a reason to omit the deploy. This dimension materialises the `deploy` row in the Verification Record, which a done-claim must carry. The **human-prereq** dimension lists any human action this slice can't proceed without — a provisioned account/credential, an enabled cloud API, an OAuth/app registration, a DNS record, a set secret/env-var (reference the `prerequisites.md` item by **name + where to set it, never the value**) — each as a **step-by-step action**, or `N/A`. A non-`N/A` human-prereq is front-loaded to `_human-input.md` **before** the task is dispatched (`autorun` parks it rather than guessing), so the build never stalls mid-slice discovering it.
3a. **These referenced IDs ARE the task's obligation contract.** The set you fix here — every `AC-`/`API-`/`EVT-`/`SEC-`/`NFR-`/… the dimensions reference — is **frozen before implementation** and is exactly what the per-task **Verification Record** is generated from at build time (`check_task_record.py --init`): the implementer is held to *this* bar and can't shrink it. So reference precisely and completely — a missing reference here is an obligation that silently won't be enforced.
4. **Force every gap to a resolution.** An UNRESOLVED gap forces one conscious choice: complete it now (elicit just-in-time → it lands in the source spec), mark `N/A — why`, or accept a default (write an ADR, reference it). For a UX/decision gap, produce a concrete default for the user to ratify, don't author blind.
5. **Tag, order, finalize.** Tag each task's phase (MVP/near/deferred) and dependencies; set its `afk:` marker. Order = topological sort, MVP-first → `build-order.md`. Apply the **acceptance gate**: each slice is independently **valuable / demoable** (delivers a user-visible outcome on its own, not a horizontal layer); record that outcome in the manifest. Implementation-final only when no dimension is an UNRESOLVED gap and the gate holds.
6. **Iterate** — derive the manifest, detect unresolved dimensions (especially slice-specific UX), ask or produce, until an agent could implement with zero ambiguity.

## Rules
- **Minimal & vertical** — the smallest independently-testable end-to-end slice; never horizontal layers, never big-bang; right-size to one agent session
- **Walking-skeleton first** — it stands up two path-scoped governance workflows: (a) **code governance** — pre-commit hooks + the app CI/CD pipeline as GitHub Actions (fill in the stack's real format/lint/type/test/fitness commands; enable CodeQL + Dependabot) **incl. a real deploy to the first environment of the ratified promotion path (per `infra-ops/topology.md`) with a smoke test + automated rollback, plus the per-hop promotion gates from `cicd.md`** — never a faked/skipped deploy; (b) **spec/doc governance** (framework-provided, ready-to-run, **project-local** — never a global hook) — install the provided `spec_governance_hook.sh` as the project's spec **pre-commit hook** (it runs `lint_spec.py` + `guard_derived.py` over `spec/`, scoped to this repo), plus the same enforcers on PRs, the docs-site deploy, branch protection, MR template; (c) **exec-order governance** (framework-provided, **project-local** — never a global hook) — run `python3 ${CLAUDE_PLUGIN_ROOT}/tools/install_exec_gates.py` — it vendors `gate_exec.py` + `check_task_record.py` into this repo's project tools folder alongside the spec-governance tools and merges the **PreToolUse exec-gate** into the project's `.claude/settings.json` (one atomic step; it refuses to wire the hook if the script isn't in place, so a half-install can't block edits). **Commit the wired `.claude/settings.json` + the vendored tools** (they belong to the repo, not a personal `settings.local.json`) so that **git worktrees inherit the gate** — AFK waves run each task in its own worktree, which only sees committed files, so an uncommitted hook silently wouldn't fire there. The gate is the tool-call-time sibling of the pre-commit hook: it enforces **red-before-green** (a `src/**` edit is blocked until a failing test was recorded for the active task) and blocks a **hollow done-claim** (flipping a task to `status: done` while its Verification Record is unmet) — the per-task build *order* a commit-time check structurally can't see. Override per call with `GRILLSPEC_GATE_OFF=1`
- **Acceptance gate at finalization** — each slice independently **valuable / demoable** with a recorded user-visible outcome; a slice that isn't demoable on its own is a horizontal cut to re-slice, not a task
- **Each task is a complete reference manifest** — every applicable dimension is a spec-ID ref, `N/A — why`, or `GAP — UNRESOLVED`; **implementation-final = no UNRESOLVED gap remains**
- **`afk:` marker** — `eligible` (an agent implements this solo, opens a green PR, and **auto-merges only where the conventions' review policy permits solo-merge**; when convention mandates ≥2-person review / branch protection on the path, the agent parks the merge as a HITL gate rather than self-approving — `eligible` never overrides the review policy) when there's no UNRESOLVED gap and no HITL trigger; else `blocked — <trigger>`. **HITL triggers (closed list):** a visual/UX decision · a product/strategy call · a legal/compliance sign-off · an external credential/access · an irreducible preference fork · **a merge the review policy reserves for a human**. **Prefer AFK** — a trigger outside this list is really an unresolved gap to resolve now
- **Coverage** — every in-scope use-case covered by ≥1 task, and every in-scope **AC owned by exactly one task** (a sibling task that builds on the work `depends-on` the owner — it never re-covers the AC; the AC is the unit of acceptance, with one Verification Record + `@covers` test); the walking-skeleton task exists and is first; build-order acyclic and MVP-first
- **module internals are JIT, not up front** — the architecture fixed the seams; a slice flags `design: design-first` only when genuinely complex (hard algorithm · real concurrency · cross-context saga), then its modules are designed & reviewed before coding; a simple/CRUD slice designs-as-it-TDDs
- **`DoD` is a reference, not a restatement** — the invariant done-bar is the per-task **Verification Record** (the standard gate rows regenerated by `check_task_record.py --init`: tests-first · tests:layers · coverage · mutation · fitness:no-fakes · fitness:architecture · spec-lint · deploy · conformance · traceability). The task's `DoD` field **points at that gate** and spells out only the **task-specific deltas** (the coverage/mutation numbers, the deploy surface, any `N/A — why`). **Never hand-restate the invariant rows** — a copy-pasted DoD line drifts and silently undersells the enforced bar (it's how conformance/deploy/fitness get dropped); the gate, not the manifest, owns the standing rows

## Output
**Stable IDs** (bare type prefix): `T-` task / vertical slice (`T-001` = the walking skeleton, always first); one file `T-NNN.md` per task.
Written under `delivery/tasks/`:

| File | Captures | Format |
|---|---|---|
| `build-order.md` | dependency order of the slices, MVP-first | ordered list or Mermaid DAG |
| `T-001.md` | the walking-skeleton task (architecture end-to-end + repo scaffold + real deploy to the first env of the ratified promotion path w/ smoke test + automated rollback) | prose |
| `T-002.md` | one task = one vertical slice; every dimension an ID ref, N/A, or resolved gap; a user-visible-outcome row | single `field \| value` table |
| `T-NNN.md` | …one file per slice (slice-specific UX inlined here; the `ux` dimension points at the slice's `JRN-` journey + prototype screen) | a single `field \| value` table, one row per dimension: behavior (one `<br>`-line per `UC-`, its `AC-`s on following lines) · domain · data · api · ux (`JRN-` + states) · a11y · security · nfr · integration · placement · design · prototype-review · tests (**one `<br>`-line per AC inside the cell**: `AC-id — tiers — intent`) · deploy · human-prereq · deps · DoD (**a reference to the standard gate + task deltas, never a restatement**), each an ID ref or N/A · plus title/phase/afk/outcome rows |

ADRs → `adr/ADR-TASK-NNN.md`
*(DERIVED & regenerate-only)*
Consumes: the use-cases / acceptance criteria, domain model, architecture (incl. the module map & seam contracts), the **API contracts** a slice builds against (the cross-boundary published language — walking-skeleton / API-first), conventions, product phasing, and the UX requirements (incl. the information architecture). The detailed data schema is **not** a task input — it's emergent during implementation, not specified up front. The per-slice clickable prototype screen is generated **at execution time**, not up front — the task's `ux` dimension references it then.

## Excludes
writing the code (execution) · release dates/sprints (the product *phasing* MVP/near/deferred orders the work, not calendar dates) · the global UX (only slice-specific UX is produced here)
