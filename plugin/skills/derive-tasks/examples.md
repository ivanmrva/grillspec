# derive-tasks — worked example

A single implementation-final task manifest — **one uniform `field | value` table**; the two sub-structured fields (`behavior`, `tests`) carry their structure **inside their cell** with `<br>` line-breaks:

```markdown
# T-014 — Pay an order (vertical slice)

| field | value |
|---|---|
| phase | MVP · afk: eligible |
| outcome | a buyer pays an order and sees a receipt |
| behavior | UC-014 pay an order<br>· AC-014a a valid payment is accepted and a receipt is shown<br>· AC-014b a zero / over-amount payment is rejected with a clear error |
| domain | AGG-Order · CMD-Pay · EVT-OrderPaid · INV-Order-total>=0 |
| data | DATA-03 (payment record; retain 7y) |
| api | API-Pay (POST /orders/{id}/payments) |
| ux | JRN-07 (pay-flow) — states: confirm-dialog · spinner · receipt · error · permission-denied → prototype: prototypes/ui/pay.html (built at exec — see prototype-review) |
| a11y | keyboard tab→confirm · focus trap in dialog · WCAG SC 2.4.7 (focus visible), SC 3.3.1 (error identified) |
| security | SEC-03 (only Owner/Billing may pay) · validate amount > 0 |
| nfr | ASR-002 (p95 < 300ms) |
| integration | PSP-Stripe (charge) |
| placement | billing context · src/billing/{api,app,domain,infra} |
| design | inline (CRUD-shaped over the aggregate — no hard algorithm/concurrency, so no design-first pass) |
| prototype-review | task-start, before code — auto-AFK (only realises the ratified JRN-07 states); HITL (afk: blocked — visual/UX decision) only if it introduces an unratified visual/UX choice |
| tests | fixture: seeded order — one `<br>`-line per AC, `AC-id — tiers — test intent`:<br>AC-014a — unit · contract · e2e — valid charge → receipt; contract on API-Pay<br>AC-014b — unit — amount ≤ 0 / over-amount rejected with a clear error |
| deploy | ships behind FLAG-pay to dev (first env of the promotion path) — or N/A — no new deployable surface |
| human-prereq | N/A |
| depends | T-002 (order aggregate) |
| DoD | standard gate — the Verification Record regenerated by `check_task_record.py --init` (tests-first · tests:layers · coverage · mutation · fitness:no-fakes · fitness:architecture · spec-lint · deploy · conformance · traceability); deltas: coverage 85% · mutation 70% (TEST-STRAT) |
```
Every dimension resolved (or N/A) → the agent can implement with zero ambiguity. Note the **ID-keyed shape**: `behavior` is one `<br>`-line per `UC-` with its `AC-`s on following `<br>`-lines; `tests` is **one `<br>`-line per AC inside the single `tests` cell** (`AC-id — tiers — intent`, never a prose blob and never a "see table below" pointer — the third segment is the *test intent*, and the line is what `check_task_record.py` reconciles against the test tree, where each AC's id must appear in its pinning test); `ux` references the journey by **`JRN-` id** + the interaction-states this slice realises; and **`DoD` is a *reference*, not a restatement** — it points at the standard gate (the per-task **Verification Record**, regenerated from the frozen refs) and spells out only the **task-specific deltas**, so a hand-authored line can never silently undersell the enforced bar (conformance, deploy, fitness, mutation all live in the gate). `design: inline` here because the slice is CRUD-shaped; a slice with a **hard algorithm · real concurrency · a cross-context saga** would be `design: design-first`, and its module internals get a JIT impl-design pass before coding. The **prototype is generated at execution time** and reviewed at the `prototype-review` gate **before code** — auto-AFK when it merely renders an already-ratified `JRN-` journey, HITL only when it introduces a new visual/UX decision. If the slice UX weren't yet defined, this task is *not* final — derive-and-ask the flow, record the `JRN-` + states here, then ship.
